## 一、Palantir 顾问的工作范式概述

Palantir 的前向部署工程师（Forward Deployed Engineer，FDE）模式是其顾问体系的核心。与传统咨询公司不同，Palantir 的顾问（FDE）通常深度嵌入客户业务一线，以"驻场陪跑"的方式推动数据能力落地，而非仅交付报告或方案文档。

### 核心角色定义：Forward Deployed Engineer（FDE）

- 同时具备工程师（能写代码、建模型）和顾问（懂业务、会沟通）双重能力
- 直接驻扎在客户业务部门，与业务方并肩作战
- 既是问题识别者，也是解决方案交付者和变革推动者

## 二、典型工作范式与动作清单

### 2.1 和业务"做朋友"：建立信任关系

- 主动融入业务场景，参与晨会、周例会、现场业务活动
- 用业务语言（而非技术语言）沟通，降低认知门槛
- 先倾听后建议，建立长期信任而非一次性交付
- 持续在场：不只在项目启动阶段出现，全程陪跑直到业务真正用起来
- 与业务人员建立私人层面的良好关系，争取成为"业务最信任的数据合伙人"

### 2.2 一线陪跑：嵌入式驻场

- 物理上就坐在业务团队旁边，而非隔离在 IT 部门
- 参与业务决策会议，在现场观察真实工作流程与痛点
- 识别业务人员"抱怨但没有正式立项"的问题——这才是最真实的需求
- 随业务节奏行动：在业务忙碌时提供即时支持，在业务空闲时一起复盘和规划

### 2.3 主动识别痛点：不等需求、主动挖掘

**Palantir 特别强调：如果仅仅被动响应需求，价值输出将极为有限。**

被动模式的局限：
- 业务方提的需求往往是"表面问题"，不是根本问题
- 业务方并不总是知道数据/技术能做什么，无法提出高质量需求
- 被动接单只能完成任务，无法引领变革

主动挖掘的典型动作：
- "影子跟踪"：跟着业务经理走一天，亲眼看他们在哪里卡住
- 每周主动走访 2~3 个业务场景，问"你现在最头疼的数据问题是什么"
- 在业务会议中留意反复被提及却未被解决的问题
- 梳理业务全流程地图，找出数据断点和决策盲区
- 提出业务尚未意识到的潜在机会点（如用数据预测库存积压风险）

### 2.4 推动变革：从工具交付到组织赋能

- 不只是交付一个 dashboard，而是推动业务改变工作方式
- 帮助业务团队建立"数据驱动决策"的工作习惯
- 协助业务方将新工具/新方法写入 SOP（标准操作程序）
- 在组织内部培育"数据倡导者"（Data Champion），实现自我延续的变革

### 2.5 数据产品孵化：从 0 到 1 的敏捷交付

- 采用 Rapid Prototype 策略：在 2~4 周内交付第一个可用的原型产品
- 原型优先于完美：先让业务能用起来，再不断迭代
- 与业务共同定义"MVP"（最小可行产品），聚焦最高价值场景
- 跟踪产品使用情况（用户登录频率、查看哪些模块），用数据驱动产品迭代方向
- 从 PoC（概念验证）到正式产品的路径清晰，避免"试点成功但无法推广"的陷阱

### 2.6 落地机制：让价值持续可见

- 设置可量化的业务影响指标（如决策速度提升 X%、人工报表时长减少 Y 小时/周）
- 定期汇报价值成果，让客户高层持续看到投入回报
- 建立用户反馈闭环，每月收集业务方对工具的评分和改进建议
- 主动管理"成功故事"：记录并传播具体案例，强化价值认知
- 与客户内部 IT 团队协作，确保数据产品的长期可维护性

## 三、核心心态与思维模式

| 心态 | 说明 |
|------|------|
| **主人翁意识** | 把客户业务当成自己的业务，对结果负责，而非只对交付物负责 |
| **工程师思维** | 用技术能力解决真实问题，不纸上谈兵 |
| **商业思维** | 理解业务目标和商业逻辑，用数据语言翻译业务需求 |
| **创业者心态** | 在不确定中快速行动，拥抱失败和迭代 |
| **服务心态** | 以"帮业务成功"为使命，而非以"完成项目"为目标 |
| **系统思维** | 看到局部问题背后的全局结构，提出根本性解决方案 |

## 四、技术与思维方法：顾问与业务的联动框架

### 4.1 嵌入式陪伴（Embedded Partnership）

- 顾问不是外部服务提供商，而是业务团队的"技术合伙人"
- 驻场时间越长，对业务理解越深，能发现的价值点越多
- 建立"顾问-业务"固定联络机制（如每周同步会、随时可微信沟通）

### 4.2 场景建模（Scenario Modeling）

- 从业务真实场景出发，而非从数据或技术出发
- 典型流程：**业务场景访谈 → 痛点地图绘制 → 关键决策识别 → 数据需求推导 → 原型设计**
- 每个场景对应一个"决策支持用例"：谁、在什么时候、需要什么数据、做什么决定
- 用场景原型（非技术方案文档）向业务展示能力，降低沟通摩擦

### 4.3 敏捷迭代（Agile Delivery）

- 以 1~2 周为迭代周期，快速交付、快速验证
- 优先做"高频 + 高价值"的业务场景，避免大而全
- 每个迭代结束后与业务方 Demo 确认，即时获取反馈
- 不追求完美，追求"持续变好"和"业务持续认可"

### 4.4 共创共担（Co-Creation & Co-Ownership）

- 业务方不只是需求提出者，也是解决方案的共同设计者
- 在 Workshop、原型评审中邀请业务关键人参与，让他们有"作品感"
- 推动业务方承担一部分数据治理和产品维护责任
- 共同制定 KPI 和成功标准，让双方利益对齐

## 五、主动赋能 vs 被动响应：核心差异对比

| 维度 | 被动响应模式 | 主动赋能模式 |
|------|-------------|-------------|
| 需求来源 | 等待业务方提 | 主动识别、挖掘 |
| 工作深度 | 完成交办任务 | 推动业务变革 |
| 价值输出 | 有限（功能交付） | 高（业务影响） |
| 与业务关系 | 甲乙方、服务关系 | 合作伙伴、共担风险 |
| 交付物 | 报告/功能/工具 | 业务能力提升、习惯改变 |
| 评价方式 | 项目是否按时完成 | 业务是否真正改变 |
| 长期价值 | 低（项目结束即结束） | 高（关系持续、复购续约）|

**核心结论：被动接需求只能交付功能，主动赋能才能创造价值。**

## 六、Palantir 成功案例中的典型模式

### 案例特征提炼（来源于公开报道与分析）

1. **军事与国防领域**：FDE 直接进驻战场指挥部，实时对接情报分析需求，做到"今天发现问题、明天出分析结果、后天改变决策"
2. **大型制造企业**：顾问与工厂运营团队并肩工作，识别出生产调度中的数据孤岛，推动跨系统数据打通，最终降低库存成本
3. **医疗健康机构**：不只是交付患者数据平台，而是帮助医院改变查房、排班的决策流程，真正影响医疗行为
4. **金融机构**：深入交易室，帮助交易员用数据平台快速识别风险，缩短决策链路

### 可复用经验

- "先做一件小事，做成功再推广"——小步快跑建立信任
- 数据能力落地必须配套组织能力建设，否则工具无人用
- 可视化和故事化是赢得业务信任的捷径
- 找到"内部冠军"（业务中有影响力的人）作为推广支点

## 七、对 cockpit_demo 顾问团队的实践建议

1. **建立驻场陪跑机制**：每月固定在业务部门驻场不少于 2 天，亲历业务流程
2. **主动痛点挖掘**：每季度开展一次业务场景访谈，整理痛点地图
3. **快速原型验证**：每个新场景优先在 2~3 周内交付可演示原型，用于争取业务认可和资源
4. **量化价值看板**：建立价值成果记录机制，定期整理"用数据节省了多少时间/成本"的故事
5. **共创工作坊**：每季度联合业务方举办一次场景共创工作坊，找出下阶段高价值方向
6. **培育业务数据倡导者**：在每个业务部门找一位对数据感兴趣的人，定期赋能，让他成为内部推广支点
