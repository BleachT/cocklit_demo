## 1. 调研目标定义

### 1.1 核心假设

本次调研需验证以下 3 个核心假设：

| # | 假设 | 验证问题 | 验证方式 |
|---|------|---------|---------|
| H1 | 用户需要"**对象级**"数据下钻（而非维度交叉）| 用户在发现异常时，是否需要查看单个 SKU / 门店 / 供应商的完整画像？ | 访谈 + 原型走查 |
| H2 | 用户需要"**关系链**"追溯（从结果到根因）| 当销售额下滑时，用户是否需要系统自动呈现原因链（天气→流量→转化→库存）？ | 情境调查 + A/B 测试 |
| H3 | 用户需要"**行动闭环**"（看完就能操作）| 用户是否会因无法在驾驶舱直接执行操作（补货/调价/推广）而感到沮丧？ | 可用性测试 + 问卷 |

### 2.2 Persona 详细画像

#### Persona A：CEO / VP（战略决策者）

| 维度 | 描述 |
|------|------|
| **典型职位** | 电商事业部 CEO / 商城 VP / 总经理 |
| **年龄范围** | 40–55 岁 |
| **技术水平** | 低（不操作系统，通过助理获取报表）|

**典型一天工作流：**
```
08:30 - 查看昨日全局营收日报（微信/邮件推送）
09:00 - 晨会：听汇报，提问"为什么昨天 XXX 下滑"
10:00 - 审批重大促销方案
14:00 - 投资人/合作伙伴会议，需要数据支撑
16:00 - 查看实时大屏（路过时）
17:00 - 审批日终复盘报告
```

**核心痛点：**
- ❌ 数据要等助理整理，无法实时获取
- ❌ 看到异常数据，无法快速了解原因
- ❌ 不同部门给的数据口径不一致
- ❌ 无法快速对比多个竞品/品类的全局态势

**期望：**
- ✅ 每天 30 秒内获得"今天公司怎么样"的答案
- ✅ 发现异常时，系统自动告诉我"是什么原因"
- ✅ 能快速看到"行动建议"

**关键决策场景：**
> "今天大促销售额比预期低 20%，我需要在 5 分钟内知道是哪个品类出问题、为什么、下一步怎么办。"

---

#### Persona B：运营总监（中层管理）

| 维度 | 描述 |
|------|------|
| **典型职位** | 电商运营总监 / 营销总监 / 增长负责人 |
| **年龄范围** | 32–45 岁 |
| **技术水平** | 中（会用 Excel、BI 工具，但不写代码）|

**典型一天工作流：**
```
09:00 - 查看昨日运营日报，重点看各渠道 ROI
10:00 - 与品类经理开周会，复盘上周营销效果
11:00 - 制定本周促销策略（需要数据支撑）
14:00 - 审查广告投放效果，调整预算分配
16:00 - 分析用户转化漏斗，找优化点
17:30 - 准备向上汇报的数据材料
```

**核心痛点：**
- ❌ 归因分析靠人工，要拉多张表交叉分析，耗时 2–3 小时
- ❌ 营销活动效果分析滞后，无法实时调整投放
- ❌ 不同渠道数据在不同系统，整合困难
- ❌ ROI 计算口径复杂，每次都要重新确认

**期望：**
- ✅ 活动效果实时可见，支持快速复盘
- ✅ 自动归因：是"人群定向"问题还是"创意质量"问题
- ✅ 跨渠道数据打通，一个页面看全貌

**关键决策场景：**
> "双十一活动还剩 4 小时，当前 ROI 不达标，我需要快速知道哪个渠道可以加预算、哪个应该停投。"

---

#### Persona C：品类经理（一线执行）

| 维度 | 描述 |
|------|------|
| **典型职位** | 品类经理 / 商品运营 / 选品负责人 |
| **年龄范围** | 25–38 岁 |
| **技术水平** | 中高（会用 Excel 高级功能，用过 BI 工具）|

**典型一天工作流：**
```
09:00 - 查看昨日负责品类的销售数据
09:30 - 查看库存预警，处理低库存 SKU
10:00 - 分析爆款商品，决定是否追加备货
11:00 - 复盘上周定价策略效果
14:00 - 选品分析：哪些新品可以引入
15:00 - 竞品价格监控，决定是否调价
16:00 - 撰写周报，汇报品类健康度
```

**核心痛点：**
- ❌ 库存预警不及时，经常断货或积压
- ❌ 调价操作需要跳转多个系统，效率低
- ❌ 商品竞争力分析依赖人工抓取竞品数据
- ❌ 无法快速了解某个 SKU 滞销的根本原因

**期望：**
- ✅ 库存预警自动推送，支持一键补货
- ✅ 在同一界面看到：销量 + 库存 + 价格弹性 + 竞品价格
- ✅ SKU 下钻：一键看到某商品的完整生命周期数据

**关键决策场景：**
> "某款车载充电器库存只剩 50 件，按当前销速 3 天后断货，我需要立刻决定是否补货、补多少、要不要同时提价。"

---

#### Persona D：数据分析师（支撑角色）

| 维度 | 描述 |
|------|------|
| **典型职位** | 数据分析师 / BI 开发 / 数据产品经理 |
| **年龄范围** | 24–35 岁 |
| **技术水平** | 高（会 SQL、Python，熟悉 BI 工具）|

**典型一天工作流：**
```
09:00 - 处理昨晚的数据报警，检查数据质量
10:00 - 响应业务方的临时数据需求
11:00 - 开发/维护 BI 报表
14:00 - 与业务方沟通需求，设计分析方案
15:00 - 数据清洗、建模
16:30 - 编写分析报告
```

**核心痛点：**
- ❌ 大量时间在取数、清洗，真正分析时间不足 20%
- ❌ 业务方频繁变更需求，报表维护成本高
- ❌ 数据血缘不清晰，出了问题难以溯源
- ❌ 临时分析需求多，无法系统化管理

**期望：**
- ✅ 低代码/无代码方式快速响应业务需求
- ✅ 数据血缘可视化，快速定位数据问题
- ✅ 自助式分析平台，减少重复取数工作

**关键决策场景：**
> "业务方反馈某个大区的销售数据有异常，我需要在 1 小时内排查是数据质量问题还是真实业务问题，并给出结论。"

### 方法 1：15 分钟闪电访谈（Lightning Interview）

**适用场景：** 项目初期，快速收集广泛的用户需求和痛点

**准备工作：**
- 提前准备问题卡片（6 个核心问题）
- 准备录音设备（征得同意后录音）
- 准备笔记模板（见下方）

**6 个核心问题模板：**

```
Q1. [工作流] "能描述一下您典型的一个工作日开始时，第一件事是什么？"
    → 目的：了解数据消费习惯的起点

Q2. [痛点] "您在分析数据时，最让您感到沮丧的是什么事情？"
    → 目的：挖掘核心痛点，不预设答案

Q3. [工具] "您现在用什么工具来查看/分析数据？"
    → 目的：了解现有工具使用情况

Q4. [频率] "您每天大概花多少时间在数据查看/分析上？"
    → 目的：量化数据工作的时间投入

Q5. [理想] "如果有一个理想的数据工具，您最希望它能帮您解决什么问题？"
    → 目的：收集未被满足的需求

Q6. [决策] "能举一个最近因为数据不够好用，而导致决策延误或出错的例子吗？"
    → 目的：收集具体故事，验证痛点真实性
```

### 方法 2：工作流跟踪（Contextual Inquiry）

**适用场景：** 深度理解用户的真实工作场景，发现用户自己没有意识到的痛点

**执行步骤：**
1. 与用户约定 1–1.5 小时，在用户实际工作环境中观察
2. 让用户完成真实工作任务，同时"出声思考"（Think Aloud）
3. 观察者只记录，不主动帮助
4. 工作结束后，进行 15 分钟补充访谈

**观察清单：**

```markdown
## 工作流跟踪观察清单

观察对象：____________  日期：____________

### 工具使用观察
- [ ] 用户使用了哪些工具？（列举）：____________
- [ ] 工具之间的切换频率：____________
- [ ] 是否有手动拷贝数据的行为？  □是 □否
- [ ] 是否使用了 Excel 中转？  □是 □否

### 数据查找行为
- [ ] 找一个数据平均需要多少步操作？____________
- [ ] 是否有"找不到"的情况？  □是 □否（记录具体场景）：____________
- [ ] 是否有"找到了但不信任"的情况？  □是 □否

### 决策行为
- [ ] 是否有"看到数据后不知道怎么行动"的情况？  □是 □否
- [ ] 决策通常需要几个数据来源的交叉验证？____________
- [ ] 是否有"等待他人提供数据"的场景？  □是 □否
```

### 方法 3：卡片分类法（Card Sorting）

**适用场景：** 验证驾驶舱的信息架构，了解用户对 KPI/指标优先级的认知

**准备材料：**

```
KPI 卡片（共 20 张）：
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  实时 GMV     │ │  订单转化率   │ │  库存周转率   │ │  营销 ROI    │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  客单价       │ │  退货率       │ │  新客占比     │ │  复购率      │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  流量来源分布  │ │  爆款商品排名  │ │  滞销商品预警  │ │  价格弹性    │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  供应商交货率  │ │  用户满意度   │ │  活动效果对比  │ │  区域销售分布 │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  毛利率趋势   │ │  竞品价格对比  │ │  用户画像分析  │ │  渠道效率对比 │
└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘
```

**执行步骤：**
1. 将 20 张卡片打乱顺序交给用户
2. 请用户按"对您最重要"到"最不重要"排序
3. 请用户将卡片分组（自由分类），并为每组命名
4. 询问用户排序/分组的理由

**分析方法：**
- 统计每张卡片的平均排名（前 5 名即核心需求）
- 用相似矩阵分析哪些指标被用户归为同类
- 对比不同角色的排序差异，发现角色需求分歧

### 方法 4：原型走查（Prototype Walkthrough）

**适用场景：** 验证驾驶舱的 4 层下钻设计（L0→L1→L2→L3）是否符合用户心智模型

**任务脚本设计：**

```markdown
## 任务 1：发现问题（L0 → L1）
"今天早上 9 点，你打开驾驶舱，发现需要关注的异常。请操作给我看。"

观察要点：
- 用户能否在 30 秒内找到最重要的异常？
- 用户的视线路径是什么？（从左到右？从上到下？）
- 用户是否理解各个模块的含义？

## 任务 2：归因分析（L1 → L2）
"你发现某个品类的销售额下滑了 15%，请找出原因。"

观察要点：
- 用户是否知道如何触发下钻？
- 用户期望看到什么样的归因展示？
- 用户能否理解"关系链"视图？

## 任务 3：执行操作（L2 → L3）
"你确认某款商品需要补货，请完成补货操作。"

观察要点：
- 用户能否找到操作入口？
- 用户对"在驾驶舱内直接操作"的接受度？
- 操作后用户期望看到什么反馈？
```

**可用性评估量表（SUS 简化版）：**

```markdown
请对以下每项评分（1=完全不同意，5=完全同意）：

2. 我认为这个系统不必要地复杂。            [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
3. 我认为这个系统很容易使用。              [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
4. 我需要专业人员帮助才能使用这个系统。    [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
5. 我认为系统中各个功能配合得很好。        [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
6. 我认为这个系统有太多不一致的地方。      [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
7. 大多数人很快就能学会使用这个系统。      [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
8. 我觉得使用这个系统很麻烦。              [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
9. 使用这个系统让我感到非常自信。          [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
10. 在使用这个系统之前我需要学习很多东西。 [ 1 ] [ 2 ] [ 3 ] [ 4 ] [ 5 ]
```

### 方法 5：A/B 方案投票（Concept Test）

**适用场景：** 比较"传统 BI 方案"与"Palantir 本体论方案"，验证用户是否接受新范式

**对比维度设计：**

| 对比维度 | 方案 A（传统 BI） | 方案 B（Palantir 本体论）|
|---------|----------------|------------------------|
| 数据呈现方式 | 固定报表 + 图表 | 对象卡片 + 关系图谱 |
| 异常处理方式 | 红色告警，人工排查 | 自动归因，呈现原因链 |
| 操作方式 | 只看数据，跳转其他系统操作 | 在驾驶舱内直接执行操作 |
| 导航方式 | 菜单导航 | 对象下钻导航 |
