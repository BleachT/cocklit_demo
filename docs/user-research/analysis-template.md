## 2. 定性数据编码

### 2.1 编码方法说明

定性编码使用 **开放性编码 → 主题聚合** 的两阶段方法：

```
第一阶段：开放性编码
  - 逐字阅读访谈记录
  - 为每个有意义的观点/行为打上标签
  - 标签要描述"是什么"，不预设类别

第二阶段：主题聚合
  - 将相似标签归为同一主题
  - 统计每个主题的出现频次
```

## 3. 需求优先级矩阵（MoSCoW）

### 3.1 MoSCoW 方法说明

| 类别 | 定义 | 产品策略 |
|------|------|---------|
| **Must Have（必须有）** | 没有它，产品无法使用 | 第一期必须实现 |
| **Should Have（应该有）** | 重要但不是紧急关键 | 第一期尽量实现 |
| **Could Have（可以有）** | 有了更好，没有也可以 | 第二期考虑 |
| **Won't Have（暂不做）** | 当前不在范围内 | 明确排除 |

---

### 3.2 需求优先级矩阵

**填写说明：**
- **支持率** = 调研中提到此需求的用户比例
- **紧迫性** = 用户对此需求的紧迫程度（基于痛点评分）
- **复杂度** = 实现难度（高/中/低，由技术团队评估）

| 需求描述 | 支持率 | 紧迫性 | 复杂度 | MoSCoW 分类 | 优先级分数 |
|---------|--------|-------|-------|------------|----------|
| 实时 GMV 等核心指标监控 | | | | | |

| 自动归因分析（关系链）| | | | | |

### 5.3 用户故事映射

> 基于调研发现，为每类角色定义关键用户故事

**CEO / VP 用户故事：**
```
作为 [CEO]，
我希望 [打开驾驶舱后 30 秒内了解今天公司的整体状态]，
以便于 [快速判断是否有需要我关注的异常]。

验收标准：
- 首屏展示最关键的 5 个健康指标
- 异常状态有醒目的视觉标识
- 点击异常可以看到系统初步归因
```

**运营总监用户故事：**
```
作为 [运营总监]，
我希望 [当促销活动效果不达标时，系统能在 5 分钟内呈现原因链]，
以便于 [快速决定是否调整投放策略]。

验收标准：
- 自动识别 ROI 异常
- 展示原因链（流量→转化→价格弹性分析）
- 每个节点有数据支撑和置信度说明
```

**品类经理用户故事：**
```
作为 [品类经理]，
我希望 [在一个页面内看到某个 SKU 的全部相关数据并直接发起操作]，
以便于 [减少在多个系统之间切换的时间]。

验收标准：
- SKU 详情页包含：库存、销量、价格弹性、竞品价格
- 页面内提供"发起补货申请"和"调价"操作入口
- 操作后能追踪执行状态
```

## 6. 用户旅程地图

### 6.1 运营总监旅程：大促数据复盘

| 阶段 | 触发事件 | 具体行为 | 思考过程 | 情绪状态 | 痛点 | 机会点 |
|------|---------|---------|---------|---------|------|-------|
| **1. 发现问题** | 大促结束第二天早上 | 打开数据系统查看 GMV | "达标了吗？" | 😰 紧张 | 数据可能还未更新 | 确保数据实时性 |
| **2. 确认数据** | 看到 GMV 不达标 | 切换多个系统核实数据 | "这个数字准不准？" | 😤 沮丧 | 数据口径不一，需要 2 小时整合 | 统一数据源 |
| **3. 归因分析** | 确认数据后 | 开始逐个排查：渠道→品类→商品 | "是哪个环节出了问题？" | 😩 疲惫 | 归因靠人工，准确率低 | 自动归因 |
| **4. 制定方案** | 初步找到原因 | 与团队讨论，制定调整方案 | "下次怎么避免？" | 🤔 思考 | 决策缺少数据支持 | 提供数据支撑的方案推荐 |
| **5. 汇报向上** | 方案确定后 | 整理 PPT，向 VP 汇报 | "怎么讲才有说服力？" | 😓 压力 | 整理数据材料耗时 | 自动生成汇报报告 |

**关键洞察：**
- 😩 情绪最低点：步骤 2（数据整合）和步骤 3（手工归因）
- ✅ 最大机会点：用自动归因替代步骤 3，节省 2–4 小时

---

### 6.2 品类经理旅程：处理滞销商品

| 阶段 | 触发事件 | 具体行为 | 思考过程 | 情绪状态 | 痛点 | 机会点 |
|------|---------|---------|---------|---------|------|-------|
| **1. 发现滞销** | 库存预警通知 | 打开库存系统查看 | "为什么卖不动？" | 😐 平静 | 预警来得太晚（已积压） | 更早的预测性预警 |
| **2. 分析原因** | 发现滞销后 | 切换到销售系统、竞品监控工具 | "是价格问题？质量问题？" | 😤 烦躁 | 需要切换 3–4 个系统 | SKU 一站式分析页 |
| **3. 决策** | 分析完成后 | 决定是否调价/促销/下架 | "怎么处理损失最小？" | 🤔 犹豫 | 缺少价格弹性数据 | 提供调价模拟工具 |
| **4. 执行** | 决策后 | 在 ERP 中填写调价申请，等待审批 | "流程这么慢..." | 😩 无奈 | 操作繁琐，审批慢 | 快速审批闭环 |
| **5. 跟踪效果** | 执行后 3 天 | 手动查看销量变化 | "起效了吗？" | 🤞 期待 | 没有自动效果追踪 | 操作后自动追踪效果 |

**关键洞察：**
- 😩 情绪最低点：步骤 2（系统切换）和步骤 4（执行繁琐）
- ✅ 最大机会点：SKU 一站式分析页 + 驾驶舱内嵌快速审批
